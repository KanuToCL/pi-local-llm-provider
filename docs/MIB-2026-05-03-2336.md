## 4. Smoking gun — `duration_ms: 6` across runs

| Boot | Model | Task ID | duration_ms |
|---|---|---|---|
| 23:32 | `gemma-4-E2B-it-GGUF` (2B) | T-957332fdfc1d | **6** |
| 23:33 | `gemma-4-E2B-it-GGUF` (2B) | T-1b87b8796702 | **6** |
| 23:36 | `Qwen3.6-35B-A3B-GGUF` UD-IQ2_M | T-765e87c9dc7b | **7** |

**Every task completes in 6-7ms regardless of model size.** Even on a 35B-A3B MoE, a real reply with HTTP roundtrip + tokens streamed should be ≥50ms. This is below GPU-warm-cache realistic floors.

**Working hypothesis (top-ranked):** `task_completed` is being emitted from `session.prompt()` settlement, but the actual reply text streams through `mapper.subscribe()` AFTER `prompt()` resolves. The audit duration is "agent loop registration time", not "model inference time". This was even called out at `src/session.ts:380-383`:

> *"We do NOT await the prompt's full resolution inside the queue lock — pi-mono streams events via subscribe() and the queue would hold the GPU lock indefinitely. Instead we await session.prompt() settlement (which pi-mono resolves when the agent loop ends)."*

If `prompt()` resolves immediately (microseconds), `taskState.kind` flips to `idle` instantly. The next inbound finds `idle`, gets accepted as a NEW task — but pi-mono's shared session is **still mid-stream from the previous request**. Possible outcomes:
1. The new prompt resets/aborts the in-flight stream → first message's reply gets silently truncated or dropped (matches observed pattern when only one of two messages is answered)
2. Streams interleave → garbled output (we don't see this — but might if the model were slower)
3. The new prompt is queued by pi-mono internally and the daemon has no visibility → "lost" messages

This contradicts the v0.2.1 BLESS-claimed serial-queue protection, because the queue's `running` window is collapsing to ~6ms instead of covering actual inference time.

## 5. Heartbeat side-evidence supports the theory

After every task, `pi-ping` source goes stale at exactly **~100s**, then dead at ~190s:
```
23:36:51  task_completed (7ms)
23:38:36  pi_stuck_suspected from=healthy to=degraded oldest_age_ms=102162  ← +1m45s
23:40:06  pi_stuck_suspected from=degraded to=dead oldest_age_ms=192190    ← +3m15s
```

Same pattern in the 23:32 run (`oldest_age_ms=100782`), and across earlier boots in our session. This consistent ~100s-to-degraded interval suggests `pi-ping` is touched **once per agent loop** — and after task_completed fires, no further pings come in until the next task starts. If the agent were truly busy streaming for tens of seconds, pi-ping should be re-touched mid-stream. It isn't.

So either:
- pi-mono's pi-ping touch is wired only at session.prompt() settlement (same place as task_completed) → no streaming-time visibility → nothing to update the heartbeat
- OR the agent loop *truly* finishes in ~6ms and the streaming we see is something else entirely (maybe a downstream ChannelEvent emission that's separate from the agent loop)

Either way, **the daemon's "is pi busy?" signal does not match user-perceived activity.**
