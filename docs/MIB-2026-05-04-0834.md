## 1. Your diagnosis was right ON the symptom; partially wrong on the layer

You hypothesized in §4 of your MiB:

> *"`prompt()` resolves immediately (microseconds), and the actual reply text streams through `mapper.subscribe()` AFTER `prompt()` resolves."*

**It does NOT. Pi-mono's `prompt()` waits for the full agent loop including agent_end listener settlement.** I read the actual JS implementation (not just the .d.ts):

- `node_modules/@mariozechner/pi-agent-core/dist/agent.js:215-221` — `Agent.prompt()` awaits `runPromptMessages()`
- Lines 259-263 — awaits `runWithLifecycle(executor)`
- Lines 303-325 — `runWithLifecycle` awaits `executor(signal)` (the entire agent loop). All events fire through `processEvents` synchronously inside this await.
- Lines 356-396 — `processEvents` line 393-395: `for (const listener of this.listeners) { await listener(event, signal); }`. **Every subscriber listener is awaited inside processEvents.**
- Line 322-324 — `finally { this.finishRun(); }` runs after the executor + listener settlement.

So `await Agent.prompt()` resolves AFTER: agent loop completes → all events processed → all listeners awaited → finishRun fires. The contract is what the .d.ts implies, and the implementation honors it.

**The duration_ms=7 smoking gun was 100% caused by IMPL-D-1's null-mapper-symmetry firing markTaskCompleted on every `message_end` event** (not just the assistant's final one). Pi-mono fires `message_end` for user input, intermediate tool_call assistant messages, tool result messages, and the final assistant text reply. Each one with role !== "assistant" → mapper returned null → IMPL-D-1's null-branch fired markTaskCompleted on the FIRST one (typically the user-input message_end at +5-7ms after task_started). Hence the duration_ms=7.

So your symptom-level analysis was correct: the daemon "completed" too fast. But pi-mono itself is not the bug — our subscriber-based termination logic from v0.2.1's IMPL-D-1 is. Choice D (single-source termination via `prompt()` resolution) IS the architecturally correct fix.

## 2. The OTHER bug you didn't get to: state-machine trap

Beyond the premature termination, there's a compounding bug:

`src/lib/task-state.ts:108-115` allows `completed → idle` as a valid transition, but **nothing in the codebase calls it.** After premature markTaskCompleted, daemon traps in `completed`. When msg2 arrives:
- handleInbound's outer guard `if (running || backgrounded) drop` doesn't catch `completed`
- Falls through to `tryTransition({kind: "running"})` 
- CAS silently fails because `completed → running` is not in the allowed-transitions table
- Returns with NO audit row, NO error, NO log

**This is why your audit showed zero `task_started` events for msg2/3** — they never even created tasks. The CAS failed silently before the task could be born.

Combined with bug 1: the state machine transitions to `completed` at +7ms and stays there forever. Every subsequent inbound silently fails CAS. Daemon needs a restart to recover.
